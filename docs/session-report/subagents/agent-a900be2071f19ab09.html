1	# Subagent Sessions — Design
2	
3	**Date:** 2026-07-12
4	**Status:** Approved
5	
6	## Goal
7	
8	Make subagent work first-class in ccwhid reports:
9	
10	1. Agent results fed back into the main session always render as a card
11	   attributed to the agent — never as a "You" card.
12	2. The transcripts of a root session's subagent sessions can optionally be
13	   included in the report as separate linked HTML pages.
14	3. When subagent sessions are included, their token usage and estimated cost
15	   roll up into the main usage summary.
16	
17	## Background: how subagent data appears on disk
18	
19	Three generations of subagent storage exist in Claude Code transcripts:
20	
21	1. **In-file sidechains** — records with `isSidechain: true` inside the root
22	   session's own `.jsonl`. Already handled by ccwhid (`--include-subagents`
23	   renders them inline inside the Task tool card).
24	2. **Linked agent-session files (modern Agent tool)** —
25	   `<projectDir>/<sessionId>/subagents/agent-<agentId>.jsonl`, one file per
26	   agent, each with a sibling `agent-<agentId>.meta.json`:
27	
28	   ```json
29	   {
30	     "agentType": "general-purpose",
31	     "description": "Implement Task 12: Profiles view",
32	     "toolUseId": "toolu_0185fny8NUzysKdpquY27j3q",
33	     "spawnDepth": 1
34	   }
35	   ```
36	
37	   Records inside the agent file are ordinary user/assistant records
38	   (they carry `isSidechain: true` and an `agentId` field). `toolUseId`
39	   links the agent to the Task/Agent tool call in the parent transcript.
40	   `spawnDepth` can exceed 1 (agents spawned by agents); all files live
41	   flat in the one `subagents/` dir regardless of depth.
42	
43	   Observed scale: a single root session with 256 agent files totalling
44	   82 MB of JSONL. This rules out inlining transcripts into one HTML file.
45	3. **Flat SDK-spawned session files** — separate top-level session files with
46	   `entrypoint: sdk-*` / `promptSource: "sdk"` (e.g. review-hook runs). They
47	   carry **no parent-session linkage** and cannot be reliably attributed to a
48	   root session.
49	
50	**Agent results in the main timeline** arrive in one of two shapes:
51	
52	- Synchronous agent: the `tool_result` attaches to the Agent/Task tool call
53	  (already rendered inside the tool card — correct today).
54	- Background agent: a `user` record whose message content is a **string**
55	  starting with `<task-notification>`, e.g.
56	
57	  ```
58	  <task-notification>
59	  <task-id>acb6584f99f2f81fd</task-id>
60	  <tool-use-id>toolu_0112a19E...</tool-use-id>
61	  <output-file>/private/tmp/.../tasks/acb6584f99f2f81fd.output</output-file>
62	  <status>completed</status>
63	  <summary>Agent "Implement Task 12: Profiles view" finished</summary>
64	  <note>...may notify more than once...</note>
65	  <result>...full agent report, markdown...</result>
66	  </task-notification>
67	  ```
68	
69	  ccwhid currently renders this as a plain user turn — the "You" card bug.
70	  The `<task-id>` value equals the `agentId` in the linked file name;
71	  `<tool-use-id>` identifies the originating tool call.
72	
73	## Decisions
74	
75	| Decision | Choice |
76	|---|---|
77	| Transcript delivery | Separate linked HTML pages, one per agent session |
78	| Usage presentation | Merged per-model table + "of which subagents" line |
79	| Flag | Reuse `--include-subagents` for both sidechains and linked agent sessions |
80	| Result card | Agent-attributed card at its timeline position |
81	
82	## Design
83	
84	### 1. Data model
85	
86	`model.Session` gains:
87	
88	```go
89	Agents []AgentSession
90	```
91	
92	```go
93	// AgentSession is a subagent session file linked to this root session.
94	type AgentSession struct {
95	    ID          string  // agentId, from the file name agent-<ID>.jsonl
96	    Description string  // from meta.json; falls back to ID
97	    AgentType   string  // from meta.json; may be empty
98	    ToolUseID   string  // links to the spawning tool call; may be empty
99	    SpawnDepth  int     // from meta.json; informational only
100	    Session     Session // the parsed agent transcript
101	}
102	```
103	
104	New turn kind and fields for the notification card:
105	
106	```go
107	const TurnAgentResult TurnKind = "agent-result" // alongside "user"/"assistant"
108	```
109	
110	`model.Turn` gains `AgentID`, `AgentStatus`, `AgentSummary string` (set only
111	for `TurnAgentResult` turns). The `<result>` body becomes the turn's single
112	text block, rendered as markdown.
113	
114	### 2. Parser: task-notification detection
115	
116	In `internal/transcript`, when a `user` record's message content is a string
117	whose trimmed text starts with `<task-notification>`:
118	
119	- Parse the `task-id`, `tool-use-id`, `status`, `summary`, and `result`
120	  elements with tolerant string scanning (no XML parser; the payload is
121	  pseudo-XML with unescaped markdown inside `<result>`).
122	- Produce a `TurnAgentResult` turn: `AgentID` = task-id, `AgentStatus` =
123	  status, `AgentSummary` = summary, blocks = one text block containing the
124	  result body. If `<result>` is missing or empty, use the summary text as
125	  the body.
126	- If parsing fails to find both `<task-id>` and `<summary>`, fall back to a
127	  plain user turn (current behavior) — never drop the record.
128	- A task-id may notify more than once; each notification renders as its own
129	  card at its timeline position (no dedup).
130	
131	This fix is **unconditional** — it applies regardless of `--include-subagents`
132	and `--usage`. Misattribution is a bug, not an option.
133	
134	### 3. Loader: linked agent sessions
135	
136	New package-level function in `internal/transcript`:
137	
138	```go
139	// LoadAgentSessions parses <dir>/subagents/agent-*.jsonl, where dir is the
140	// transcript path minus ".jsonl". Returns them sorted by file name.
141	func LoadAgentSessions(transcriptPath string, opts Options) ([]model.AgentSession, error)
142	```
143	
144	- Called from `run.go` only when `--include-subagents` is set.
145	- Each agent file is parsed with the existing `Parse`. Inside an agent's own
146	  file its records are the main chain, even though they carry
147	  `isSidechain: true`; the parser must treat them as main-chain records when
148	  parsing an agent file. Concretely: `Options` gains `AgentFile bool`; when
149	  set, `IsSidechain` on records is ignored.
150	- Metadata from `agent-<ID>.meta.json`; a missing or unreadable meta.json is
151	  tolerated (Description falls back to ID, other fields zero).
152	- A missing `subagents/` dir yields an empty slice, no error. An unparsable
153	  agent file is skipped with a warning to stderr, not a fatal error.
154	
155	### 4. Renderer: agent-result cards and subagent pages
156	
157	**Agent-result card** (always): a `TurnAgentResult` turn renders as a card
158	with role label `Agent` plus the summary (e.g. `Agent · Implement Task 12:
159	Profiles view`), a status badge (`completed` / other), and the result body
160	as markdown. Distinct accent styling from You/Claude cards. Search text and
161	prompt-jump behavior treat it like an assistant card (it is not a prompt).
162	
163	**Subagent pages** (when agent sessions are loaded):
164	
165	- Output layout: `outDir/subagents/agent-<ID>.html`, sharing
166	  `outDir/assets/` via relative paths (`../assets/...`).
167	- Each page uses the same report template: title = agent description,
168	  subtitle shows agent type and ID, plus a back-link to `../index.html`.
169	- Each page gets its own usage card when `--usage` is set (computed over
