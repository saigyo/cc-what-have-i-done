The problem

DecodeProjectDir reverses the /→- encoding by replacing every - with /. This is LOSSY when a real path segment contains a hyphen (e.g. the project cc-what-have-i-done lives at ~/IdeaProjects/cc-what-have-i-done, encoded as -Users-[user]-IdeaProjects-cc-what-have-i-done; naive decode wrongly yields ~/IdeaProjects/cc/what/have/i/done). Claude Code's on-disk directory name is genuinely ambiguous, so decoding alone cannot recover the true path.

The fix: use the real cwd from the transcript

Every user/assistant record in a transcript carries a "cwd" field with the true absolute path. Use it as the authoritative project path; fall back to the lossy decode only when no cwd is present.
